# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Static marketing site for Tuncerdesing (industrial kitchen equipment / restaurant setup company, Istanbul). No build system, no package manager, no bundler â€” plain HTML/CSS/JS served as-is (deployed via GitHub Pages, per `CNAME` pointing to `tuncerdesing.com`).

## Development

There is no build/lint/test tooling in this repo. To preview changes, open a page directly in a browser or serve the directory with any static file server (e.g. `python -m http.server`). There are no automated tests.

## Architecture

- **Multi-language via real per-language pages, not client-side swapping.** Turkish is the default language and lives at root URLs (`index.html`, `endustriyel-mutfak-ekipmanlari.html`, `blog/index.html`, etc.). English, French and Arabic are full static translations at `en/`, `fr/`, `ar/`, mirroring the same file/folder structure one level down (e.g. `en/index.html`, `en/blog/mutfak-ekipmani-maliyeti-nasil-hesaplanir.html`). Every page's `<head>` carries a full `hreflang` set (tr/en/fr/ar/x-default) plus a self-canonical. Arabic pages set `<html lang="ar" dir="rtl">`. **There is no runtime i18n system** (no `js/language.js`, no `lang/*.json` fetching) â€” each language's text is baked directly into its own HTML file. When editing user-facing copy, you must edit each of the 4 language files for that page individually; there is no single source of truth to edit once and propagate.
- `lang/*.json` (tr/en/fr/ar) still exist as **reference translation strings only** â€” they were the source used to originally generate the translated homepage copies, kept around as a convenient lookup for consistent terminology, but nothing in the site loads them at runtime. Don't assume they're wired up to anything.
- `index.html` (and its `en/fr/ar` counterparts) is a single-page layout: hero, stats, services, process, about, CTA, FAQ, contact, footer are all `<section>`s in one file. Styling is Tailwind via CDN (`cdn.tailwindcss.com`, not a compiled build) plus custom overrides in `css/style.css`. Animations use GSAP + ScrollTrigger (also via CDN), loaded with `defer` alongside `js/main.js` â€” order matters (GSAP/ScrollTrigger must execute before `main.js`, which calls `gsap.registerPlugin(ScrollTrigger)` at top level), so keep all four `defer` scripts in that relative document order if touching `<head>`/end-of-`<body>` script tags.
- **Service pages**: 6 services (`endustriyel-mutfak-ekipmanlari.html`, `mimarlik-destegi.html`, `ozel-uretim-mutfak-ekipmanlari.html`, `restoran-tadilat-insaat.html`, `tabela-gorsel-tasarim.html`, `teknik-destek-bakim.html`), each with TR/EN/FR/AR versions (24 files total), same shared template (nav, breadcrumb, language switcher row, hero, feature grid, CTA, footer, WhatsApp float). No JS framework â€” these are hand-authored static pages, not generated from a template engine, so a content change must be applied to each of the 4 language files by hand.
- **Blog**: `blog/` (TR) + `en/blog/`, `fr/blog/`, `ar/blog/`, each with an `index.html` listing page and 10 article pages (44 files total), same template pattern as service pages (breadcrumb, language switcher, `BlogPosting` JSON-LD).
- **Language switcher pattern**: service/blog pages use a plain text row (`TR / EN / FR / AR`) linking to sibling files across `../en/...`, `../fr/...`, etc. The homepage nav uses a fancier flag dropdown (`#languageButton` / `#languageMenu`), but its links are also plain navigation to `en/index.html` / `fr/index.html` / `ar/index.html` â€” not an AJAX swap. The dropdown open/close toggle is handled in `js/main.js` (guarded with null-checks so it's a no-op on pages without that markup).
- `gizlilik-politikasi.html` (privacy policy) and `kullanim-kosullari.html` (terms of use) are Turkish-only standalone pages, linked from every page's footer with relative paths adjusted for directory depth (`../gizlilik-politikasi.html` from one level down, etc.). They have not been translated.
- `js/main.js` â€” all page interactivity: GSAP scroll-reveal animations, counter animations, navbar scroll state, mobile menu toggle, language dropdown toggle, in-page smooth-scroll for `a[href^="#"]` links (uses `preventDefault` + `scrollIntoView`), and the contact form submit handler (AJAX POST to Formspree, see below). Shared by all homepage variants; service/blog pages don't include it (they don't need the animations/form it drives).
- Contact form (`#contactForm`, homepage only) posts to a Formspree endpoint (`action="https://formspree.io/f/..."`); submission is intercepted in `main.js` and sent via `fetch`, showing the `#toast` element on success.
- Cookie consent banner (`#cookieBanner`) and its accept/reject handlers are defined inline at the bottom of each homepage variant (not in `js/main.js`); state persists via `localStorage` (`cookieConsent` key). Banner copy is translated per language, not shared.
- Analytics/tracking: Google tag (gtag.js) and Meta/Facebook Pixel are loaded inline in every page's `<head>`, same IDs across all languages (single shared business, not per-market tracking).
- SEO: every page has Open Graph/Twitter meta, a `BreadcrumbList` JSON-LD (service/blog pages) mirroring the visible breadcrumb, plus page-appropriate `LocalBusiness`/`Organization`/`WebSite`/`Service`/`BlogPosting`/`FAQPage` schema. `llms.txt` at the site root summarizes the business for AI agents/crawlers. Keep schema in sync if business info (address, phone, hours) changes â€” it's duplicated per language file.
- `robots.txt` / `sitemap.xml` â€” sitemap lists all TR/EN/FR/AR URLs (homepage, 6 services Ă— 4 languages, blog index + 10 posts Ă— 4 languages).
- Favicon/app icons live in `images/` (`favicon-16x16.png`, `favicon-32x32.png`, `apple-touch-icon.png`) plus `site.webmanifest` at root â€” referenced with root-relative paths (`/images/...`, `/site.webmanifest`) so they resolve correctly regardless of a page's directory depth.
- Performance: Font Awesome and Google Fonts CSS are loaded via the `preload` + `onload` swap-to-stylesheet pattern (non-render-blocking) with a `<noscript>` fallback, across every page. The Tailwind CDN `<script>` itself is intentionally **not** deferred â€” the whole site's layout depends on Tailwind's runtime JIT compiler, so deferring it causes a visible flash of unstyled content. Moving off the Tailwind Play CDN to a compiled/purged stylesheet is a known, deliberately-deferred future optimization (would need a one-time local build step via `npx tailwindcss`, since there's no build system currently).

## Conventions in this codebase

- Primary brand colors are CSS custom properties `--primary` (`#1e3a5f`) and `--accent` (`#c9a962`), also hardcoded as Tailwind arbitrary values (`text-[#1e3a5f]`, `bg-[#c9a962]`) throughout every page â€” both must be updated together if rebranding.
- Content is Turkish-first; when adding a new page or section, write the Turkish version first, then produce EN/FR/AR as separate hand-translated files at the corresponding `en/`, `fr/`, `ar/` path â€” there is no automated translation propagation.
- Images are loaded from Unsplash by URL (no local image assets besides the favicon/logo files in `images/`); above-the-fold images use `fetchpriority="high"` (no `loading="lazy"`), below-the-fold images use `loading="lazy"`.
- Internal links between pages should always include `target="_self"` â€” a stray `<base target="_blank">` left in `<head>` (a historical artifact) means links without an explicit target open in a new tab.
